Design Principles
How we think about design
These principles guide every decision in Stripe's design process — from the shape of a button to the information architecture of a complex flow.
Principle 01
Clarity over cleverness
Financial products require absolute clarity. Users must understand what's happening with their money at all times. We never sacrifice comprehension for visual interest. The simplest solution that communicates clearly is almost always the right one.
DoUse plain language for payment states. "Payment succeeded" not "Transaction completed successfully."
Don'tUse ambiguous icons or metaphors for financial states that require precise communication.
Principle 02
Infrastructure feels invisible
The best infrastructure is the kind you don't notice. Our UI should surface insights and actions — not itself. Components should feel like natural extensions of the user's workflow, not tools they have to learn.
DoSurface relevant data at the moment it's needed. Show fraud alerts inline with the transaction.
Don'tAdd UI chrome, decorative elements, or features that don't serve an immediate user need.
Principle 03
Trust is earned in details
Trust isn't declared — it's demonstrated through consistent behavior, precise alignment, reliable interactions, and honest communication. Every pixel inconsistency or unclear error message erodes the confidence a user places in us with their financial operations.
DoShow exact amounts, precise timestamps, and complete status information. Never approximate financial data.
Don'tRound numbers in financial contexts, use vague relative times ("a while ago"), or hide error details.
Principle 04
Global scale, local feel
Stripe operates in 135+ countries. Our design system must work across cultures, scripts, and contexts without losing coherence. Localization is not an afterthought — it's built into the foundation. RTL, international number formats, and currency representations are first-class concerns.
DoDesign layouts that accommodate LTR and RTL. Use locale-aware number and currency formatting throughout.
Don'tHardcode currency symbols, assume left-to-right reading, or use culturally specific idioms in copy.
Principle 05
Progressive disclosure
Stripe's product is powerful and complex. We reveal complexity progressively — starting with what 80% of users need, then offering depth for advanced use cases. The API shows everything; the dashboard shows what matters. Power users can always go deeper.
DoPut the most common action first. Advanced settings live in secondary panels. "Advanced options" collapses non-essential fields.
Don'tFront-load complexity. Every setting shown by default is a setting the user has to process and reject.
Principle 06
Developer experience is user experience
Stripe's primary users are developers. Code samples are UI. API responses are content. Documentation is design. The DX principles of clear naming, predictable behavior, and helpful error messages apply to the visual product too.
DoUse the same terminology in the UI as in the API. If the API has `payment_intent`, the dashboard says "payment intent."
Don'tCreate marketing-speak names in the UI that diverge from the underlying technical model. This creates confusion for developers.
Voice & Tone
How Stripe communicates
Stripe writes like a knowledgeable colleague — smart, direct, and respectful of your time. The voice is consistent; the tone adapts to context.
Core Voice Attributes
Clear, not dumbed-down
We respect our users' intelligence. Jargon is used only when it's precise and familiar. Plain language wins — but we don't over-explain.
Direct, not terse
Get to the point. No filler phrases like "Please note that..." or "We wanted to let you know..." Say what you mean, then stop.
Optimistic, not cheerful
We believe in builders. The tone is grounded and encouraging — not bouncy or exclamation-heavy. "Start building" not "Let's go! 🚀"
Expert, not boastful
We have deep expertise. We share it plainly — no inflated claims, no buzzwords. "99.999% uptime" not "industry-leading reliability."

Tone by Context
🎉 Success
Brief celebration + confirmation. "Payment received. A receipt is on its way to jan@example.com."
⚠️ Warning
Calm, factual, never alarming. "Your payout is on hold. Complete identity verification to continue."
❌ Error
Specific cause + clear next step. Never blame the user. "We couldn't process this payment. The card's billing zip code didn't match."
🔒 Security
Serious, transparent, factual. "We've locked your account after unusual activity. Contact support to restore access."
📚 Onboarding
Welcoming, clear, momentum-building. "You're almost ready to accept payments. Add your bank account to start receiving payouts."
